# 可自托管的开源网易云车机播放器 — 设计文档

- 日期:2026-07-21
- 状态:已通过设计评审,待用户复核 spec
- 代号:tesla-music(名字可改)

## Context(背景与动机)

用户在日本使用特斯拉,想在车机自带浏览器上打开一个**定制版网易云音乐**,开车时显示**滚动歌词**。

需求在讨论中演进过两次,最终定型:

1. 最初设想"只有我自己用" → 后改为**开源、可自托管**:任何人都能**一条命令部署自己的小服务器**、登录**自己的**网易云账号。
2. "解锁灰色歌曲"从必需项降级为**可选、默认关闭**的模块。

这次演进的关键结论:项目从"个人工具"变为"开源自托管产品",但通过**自托管、各用各的**分发模型,把法律与安全风险控制在与现存同类开源项目(YesPlayMusic 等)相当的水平——作者只是代码作者,不运营中心服务,不持有任何他人凭证。

## Goals(目标)

- 一个开源、可自托管的网易云 Web 播放器,针对**特斯拉车机浏览器**优化。
- 核心体验:登录自己的账号 → 浏览歌单/搜索 → 播放 → **逐行滚动歌词**(招牌功能)。
- **一键部署**:`docker compose up` 即可在自己的小服务器/VPS/家用机上跑起来。
- 车机友好:大按钮、高对比、行驶中零交互(自动连播)、被动展示为主。

## Non-Goals(明确不做)

- **不做中心托管服务**。绝不架一台服务器让所有人来登录——那会让作者成为服务运营者并持有他人网易云凭证(法律与安全灾难)。
- **不做跨平台音源替换**(从 QQ/酷狗/咪咕等搜同名歌注入 URL)。匹配易错、音质参差、法律风险最高;车机场景下放错版本比跳过更糟。
- **v1 不做多用户隔离**。单实例单账号:谁扫码谁就是这台机的用户。多用户是 v2。
- 不做原生 App、不做离线下载、不做歌曲上传/云盘管理。

## 法律与风险姿态(pivot 后核心)

| 维度 | 采取的姿态 |
|---|---|
| 分发模型 | **自托管,各用各的**。开源代码 + 一键部署,每人跑自己的实例、登己账号 |
| 作者身份 | 仅代码作者,不运营服务,不持有任何他人 cookie/凭证 |
| realIP 解锁 | **可选、默认关闭**(`UNLOCK_ENABLED=false`),开启与否与责任归**部署者** |
| 音源替换 | 完全不内置 |
| 参照先例 | YesPlayMusic、各 NCM API fork 均以此形态长期存在;原版 `Binaryify/NeteaseCloudMusicApi` 因 DMCA 于 2024 年归档,故后端改用维护中的 fork |

## 架构

三层,单 Docker Compose 编排:

```
┌─────────────────────────────────────────────┐
│  特斯拉车机浏览器 (Chromium)                   │
│  ── 前端:Vite + React + TS(静态产物)        │
│     · 扫码登录 UI                             │
│     · 歌单 / 搜索 / 播放器                     │
│     · 滚动歌词组件(LRC 时间轴同步)            │
└───────────────┬─────────────────────────────┘
                │ HTTP(同源 /api 反代)
┌───────────────▼─────────────────────────────┐
│  后端:维护中的网易云 API fork (Node)          │
│     · 扫码登录、weapi/eapi 加密签名            │
│     · realIP 注入(默认关闭)                  │
│     · 歌单/搜索/播放URL/歌词 等接口            │
│     · 单实例单账号:cookie 存于本实例会话      │
└───────────────┬─────────────────────────────┘
                │
┌───────────────▼─────────────────────────────┐
│  网易云官方 API(公网)                        │
└─────────────────────────────────────────────┘
```

### 组件职责

- **前端(React SPA)**:纯展示与交互;不碰任何网易云加密逻辑,只调本地 `/api/*`。
- **后端(NCM API fork)**:所有网易云交互、加密、cookie、realIP。前端与网易云之间的唯一桥梁(解决 CORS + 签名 + 鉴权)。
- **反向代理 / 静态服务**:让前端静态资源与 `/api` 同源,消除 CORS 与跨域 cookie 问题(nginx 或后端自带静态服务,择一)。

### 数据流(播放一首歌)

1. 前端请求 `/api/song/url`(或等价接口)带 songId。
2. 后端携带本实例的登录 cookie、按需注入 realIP,签名后转发网易云。
3. 后端把播放 URL 返回前端;前端 `<audio>` 播放。
4. 前端并行请求 `/api/lyric`,拿 LRC(及可选 `tlyric`/`yrc`),解析时间轴,随 `audio.currentTime` 滚动高亮。

## 关键技术设计

### 登录(扫码)与登录保持
- UI 展示二维码 → 手机网易云 App 扫码 → 轮询登录态 → 后端保存 cookie。
- 单实例单账号:cookie 持久化于**部署者自己的**服务器。
- 不做账号密码登录(降低风险、避免存密码)。

**登录保持(关键设计)**:cookie 存在服务器,不存在车机浏览器。
- 后端把登录成功后的 cookie(`MUSIC_U` 等长效 token,有效期通常数周~数月,使用时续期)**持久化到磁盘/Docker 数据卷**,进程重启、容器重建都不丢。
- 后端在每次调用网易云接口时**自动注入**这份 cookie,车机浏览器**不持有任何凭证**,只调本地 `/api`。
- 因此即便特斯拉浏览器激进清理 cache/cookie/localStorage,登录态也**不受影响**——它活在你自己的服务器上。**扫一次码,之后长期免登录**;仅 token 过期或主动登出才需重扫。
- 注意:多数 NCM API fork 默认是**无状态**的(把 cookie 返回给客户端、要求客户端回传)。本项目需在后端加一层**薄持久化**:登录成功后落盘保存 cookie,并在转发时自动带上——这是相对 fork 原始行为的主要增量。

### realIP 解锁模块(可选,默认关闭)
- 由环境变量 `UNLOCK_ENABLED`(默认 `false`)控制;开启时对相关接口注入国内 `realIP` 参数,绕过**海外地理围栏**(依赖部署者自己的 VIP 版权)。
- 对**真·无版权灰色歌**无效(那需要音源替换,已列为 Non-Goal)。
- README 明确:开启与否及后果由部署者自负。

### 滚动歌词
- v1:解析 LRC 时间戳,按 `currentTime` 逐行滚动 + 当前行高亮;缺歌词时优雅降级。
- v2 增强:逐字 `yrc`(卡拉OK字级高亮)、翻译行 `tlyric`。

### 特斯拉车机约束(阶段 0 在真车验证)
- 行驶中浏览器是否可用、`<audio>` 能否走车载扬声器、自动播放是否需先一次点击、屏幕/浏览器是否会被切走。
- 缓解:首次进入用一次用户点击"解锁"音频上下文;自动连播减少行驶中交互;必要时用无声保活/唤醒策略。

## 部署

- **主路径:Docker Compose 一键起**(后端 API + 静态前端 + 反代)。目标是"自己的小服务器/VPS/家用机"上 `docker compose up` 即可。
- 次要可选:Vercel/Railway 一键部署按钮。
- 配置项(环境变量):`UNLOCK_ENABLED`(默认 false)、realIP 值、端口等。
- 日本网络无墙,官方 API 公网可达,预期免代理直连即可。

## 交付阶段

- **阶段 0 — 打通验证(先做)**:后端起 + Docker 部署 + **一屏验证页**,端到端播放**一首真实歌曲**并滚动其歌词 → **真车实测**。此屏一次验证全部高风险环节(网络通路、登录+realIP 能否拿到播放 URL、音频走扬声器、歌词同步、屏幕保活)。跑通=剩下全是堆 UI;跑不通=只花一天。
- **阶段 1 — v1 完整体**:扫码登录 UI + 歌单/每日推荐/搜索 + 完整播放器(播放/暂停/上下曲/进度)+ 逐行滚动歌词 + 车机优化 UI + Docker Compose 一键部署 + README。
- **阶段 2 — 增强(可选)**:逐字 `yrc` 卡拉OK歌词、翻译行、车机深色极简主题、多用户(如需)。

## 验证方式

- **阶段 0 验证页**在真实特斯拉车机上跑通:能联网、能出声、歌词能滚、播放时屏幕不被切走。
- 后端接口:本地 `curl` 验证登录态、播放 URL、歌词接口返回正常。
- 前端:本地 dev 跑通登录→搜索→播放→歌词滚动完整链路。
- 部署:干净环境 `docker compose up` 一次成功,浏览器访问即用。
